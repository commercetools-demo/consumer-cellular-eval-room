Flexibility: Extend the Platform, Don't Fork It

The concern we hear most from teams replacing a legacy monolith, or considering the commerce module bundled with a billing suite: "A SaaS platform can't be flexible enough for what we do." The commercetools answer is API Extensions: synchronous hooks that fire at any commerce event and return custom logic inline. Your new storefront connects via REST or GraphQL. Your MVNO activation, device compatibility checks, AmDocs (or chosen billing) handoff, and AARP / USAA partner logic run in your own infrastructure. The platform is the backbone, not the ceiling.

Buy the core. Build everything that's unique to Consumer Cellular.

Tailored Questions - Flexibility

API Extensions are synchronous webhooks that fire at designated commerce lifecycle events: before or after a cart update, before order creation, on customer authentication, on price calculation, and more. Your extension receives the full event payload, runs custom logic in your infrastructure, and returns a modified response - which the platform applies inline before returning to the caller.

For the stacked-promotion scenario: when the cart updates, an API Extension fires. Your service checks AARP eligibility, USAA membership, AutoPay status, and the multi-line count, applies each discount with its proper stacking precedence via Direct Discounts and Cart Discounts, enforces caps and exclusions, and returns the priced cart. The platform accepts the result and presents the configurator-style summary to the customer. The promotion rule changes when the partner mix changes - not the platform.

Extensions are configured in Merchant Center or via the Extensions API. They point to any HTTPS endpoint - a Lambda, a Cloud Run function, or a service in Consumer Cellular's existing infrastructure. Consumer Cellular owns the logic; commercetools provides the hook.

Tailored Questions - Device Launch & Pre-Orders

A pre-order in commercetools is a normal Order created against an InventoryEntry that has a reservation against expected stock - not against on-hand stock. The Product Variant carries an "availableForPreOrder" flag (a Custom Field), a launch window, a per-customer cap, and a reservation pool size. When a customer commits in the configurator, an API Extension validates eligibility, reduces the reservation pool atomically, and creates the Order in a custom state called "PreOrder-Confirmed." No physical inventory has been picked or shipped; the platform is the system of record for the reservation.

On launch day, when physical stock lands at the carrier or warehouse, your fulfillment system publishes an Inventory event. A Subscription fires; the matching pre-orders advance from "PreOrder-Confirmed" to "Order-In-Fulfillment" and activation kicks off downstream. The customer sees a clean status path the whole way through.
