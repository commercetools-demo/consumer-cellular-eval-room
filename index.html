Flexibility: Extend the Platform, Don't Fork It

The concern we hear most from teams on custom engines: "A SaaS platform can't be flexible enough for what we do." The commercetools answer is API Extensions — synchronous hooks that fire at any commerce event and return custom logic inline. Your frontend connects via REST or GraphQL. Your business logic runs in your infrastructure. The platform is the backbone, not the ceiling.

The Scenario
Carparts.com's product team wants to launch a tiered loyalty program: customers earn points per order, points unlock a pricing tier, and a new checkout step lets them apply points as partial payment. In the current custom engine, this is a 5–8 week sprint across catalog, cart, checkout, and payments — all on commodity infrastructure. The question: how does this work in commercetools, and how much of it is configuration vs. custom code?
Tailored Questions — Flexibility

API Extensions are synchronous webhooks that fire at designated commerce lifecycle events: before or after a cart update, before order creation, on customer authentication, on price calculation, and more. Your extension receives the full event payload, runs custom logic in your infrastructure, and returns a modified response — which the platform applies inline before returning to the caller.

For the loyalty scenario: when a customer adds a product to cart, an API Extension fires. Your loyalty service checks the customer's point balance and tier, applies the appropriate price override via a Direct Discount object, and returns the enriched cart. The platform accepts this as a valid cart update. No platform modification, no fork, no platform release cycle dependency.

Extensions are configured in Merchant Center or via the Extensions API. They point to any HTTPS endpoint — a Lambda, a Cloud Run function, or a service in your existing infrastructure. Carparts.com owns the logic; commercetools provides the hook.

commercetools is fully headless. The platform exposes REST and GraphQL APIs. There is no bundled frontend, no required template, no storefront dependency. Your existing frontend — whatever stack it runs on — queries the commercetools APIs directly. Cart, checkout, search, product display, customer authentication: all driven through APIs your frontend calls.

In practice, the most common migration pattern is replacing the commerce backend incrementally. The Carparts.com frontend continues to serve traffic; the API calls that previously hit your custom engine are rerouted to commercetools endpoints one flow at a time. The customer sees no change. Your engineers flip one integration at a time and validate before moving to the next.

commercetools also provides a Frontend SDK and Composable Commerce Starter for teams that want to accelerate frontend development — but it is never required.

Configuration handles the model; your code handles only the proprietary calculation.
